GC与日志分析

GC日志

参数

  • -XX:+PringGCDetails

打印详细的gc日志

  • -XX:+PrintGCTimeStamps

打印每次出来GC发生的时间

  • -XLoggc:gc.log

设置将GC写入一个磁盘文件

案例

YoungGC案例

参数设置

-XX:InitialHeapSize=10485760 -XX:MaxHeapSize=10485760
设置初始堆大小和最大堆大小为10MB

-XX:NewSize=5242880 -XX:MaxNewSize=5242880
设置新生代大小和最大新生代大小5MB，老年代大小计算出为5MB

-XX:PretenureSizeThreshold=10485760
设置大对象阈值为10MB

运行步骤

  1. byte[] arr1 = new byte[1024 * 1024]

首先会在Eden区放入1MB对象，同时在mian线程的虚拟机栈中压入main()方法的栈帧。在mian()方法的栈帧内部，会有一个arr1变量，指向堆内存的Eden区数组

  1. arr1 = new byte[1024 * 1024]

在Eden区创建第二个数组，并让局部变量指向第二个数组，然后第一个数组没人引用了

  1. arr1 = new byte[1024 * 1024]

在Eden区创建了第三个数组，同时arr1指向第三个数组，前两个变成垃圾对象

  1. arr1 = null

arr1对象什么都不指向，前三个数组全部变成垃圾对象

  1. byte[] arr2 = new byte[ 2 * 1024 * 1024]

分配一个2MB数组，尝试放入Eden区。由于Eden区总共4MB，放不下，就会触发Young GC

动态年龄案例

参数设置

Java堆总大小20MB，新声代10MB，Eden区8MB，每个Survivor1MB，老年代10MB。
大对象超过10MB进入老年代
对象年龄超过15岁进入老年代

运行步骤

  1. 连续创建3个2MB数组，将局部变量arr1设置为null

  1. byte[] arr2 = new byte[ 128 * 1024]

在Eden区创建128KB数组，同时由arr2变量引用

  1. byte[] arr3 = new byte[ 2 * 1024 * 1024]

尝试在Eden区分配一个2MB的数组。此时算上着2MB，Eden区大小超过8MB，触发一次YoungGC

对应日志：ParNew: 8120K->780K(9216K), 0.0008585 secs

eden space 8192K, 26% used
from space 1024K, 76% used

Young GC过后，Eden区存在2MBarr3，S1区存在128KBarr2

此时Survivor Form区的对象1岁

  1. 继续分配3个2MB数组，一个128KB数组

  1. byte[] arr4 = new byte[ 2 * 1024 * 1024]

尝试放入2MB数组，触发Young GC

Full GC案例

参数设置

-XX:NewSize=10485760 -XX:MaxNewSize=10485760 -XX:InitialHeapSize=20971520
-XX:MaxHeapSize=20971520 -XX:SurvivorRatio=8 -XX:MaxTenuringThreshold=15
-XX:PretenureSizeThreshold=3145728 -XX:+UseParNewGC -XX:+UseConcMarkSweepGC
-XX:+PrintGCDetails -XX:+PrintGCTimeStamps -Xloggc:gc.log

GC日志

运行步骤

  1. byte[] arr1 = new byte[4 * 1024 * 1024]

     arr1 = null

直接分配一个4MB的对象，是大对象，直接进入老年代

  1. byte[] arr2 = new byte[ 2 * 1024 * 1024];

     byte[] arr3 = new byte[ 2 * 1024 * 1024];
 byte[] arr4 = new byte[ 2 * 1024 * 1024];
 byte[] arr5 = new byte[ 128 * 1024];

连续分配4个数组，3个2MB数组，1个128KB数组。全部进入Eden区

  1. byte[] arr6 = new byte[ 2 * 1024 * 1024];

又分配了2MB区域，发现Eden区放不下，触发一次Young GC

ParNew (promotion failed): 7430K->8248K(9216K), 0.0028134 secs
由于存活对象放不下Survivor，需要全部进入老年代，但是老年代加上目前的数据也超出10MB。
因此触发了CMS垃圾回收器的Full GC。

[CMS: 8194K->6922K(10240K), 0.1267533 secs] 11526K->6922K(19456K), [Metaspace: 2818K->2818K(1056768K)], 0.1626242 secs]
