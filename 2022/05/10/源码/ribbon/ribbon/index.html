一、寻找ribbon入口

ribbon初探，在RestTemplate用@LoadBalance注解标记，即可以使用负载均衡器进行服务调用。

从LoadBalance入手

发现LoadBalance是spring-cloud-commons项目中的注解。
spring如果和其他技术整合，特点是一定会有一个XXAutoConfiguration，去初始化整合技术关键类
从loadbalance包中寻找到两个配置文件AsyncLoadBalancerAutoConfiguration和 LoadBalancerAutoConfiguration，推测Async为后者的拓展，名字中看出是用于异步负载均衡

LoadBalancerAutoConfiguration

这个配置文件中有几个值得注意的点

  1. List<RestTemplate> restTemplates创建了一个空的RestTemplateList，推测自己初始化的restTemplate对象会放入这个list用于后边包装扩展
  2. SmartInitializingSingleton对上述的list进行了smart初始化
  3. RestTemplateCustomizer对RestTemplate设置了Interceptors
  4. 后边还出现一个RetryRestTemplate，看名字是和重试相关，目前只关注普通的

二、LoadBalancerInterceptor

上面配置文件对RestTemplate设置了拦截器，然后就看一下拦截器是实现什么功能

发现intercept接口返回response，那么拦截器就是对请求处理之后，进行了请求获取了response。
推测：

  • request request中肯定包含原本请求连接http://ServiceA/sayHello
  • body body中包含请需要的一些参数，比如json串
  • execution 底层http通讯组件

相当于不再调用RestTemplate原本的请求，而是使用intercept中的请求处理

四、获取LoadBalance

整个执行方法，其中getLoadBalancer为根据serviceId获取对应的LoadBalance。

每个要调用的服务名称，都也对应着一个spring的ApplicationContext容器，ServiceA对应着一个独立的spring的ApplicationContext容器。
在SpringClientFactory里面，一个服务对应着一个独立的ApplicationContext，里面包含了自己这个服务独立的一堆组件。如果要获取一个服务对应的LoadBalance，其实就是在服务对应的ApplicationContext中获取LoadBalance。根据ILoadBalancer接口类型，获取接口对应的实例化bean。
默认得到的负载均衡器为：ZoneAwareLoadBalancer

1.初始化serverList

  1. 在ZoneAwareLoadBalancer查找对象属性，发现在父类中存在volatile ServerList serverListImpl，这个ServerList的初始化在构造方法中。
  2. 也就是在构造ZoneAwareLoadBalancer的时候传入
  3. 回到RibbonClientConfiguration中，发现ZoneAwareLoadBalancer构造的时候，ServerList是每个@bean注入的。
  4. 就在RibbonClientConfiguration的同级中发现一个ribbonServerList，这个就是默认的不在eureka环境中的ServerList
  5. 但是在eureka整合中的ServerList不在此处。推测应该是eureka和ribbon整合包的配置中应当存在。
  6. 在spring-cloud-netflix-eureka-client工程中发现，ribbon与eureka的整合在此处存在一个configuration：org.springframework.cloud.netflix.ribbon.eureka.EurekaRibbonClientConfiguration

2.初始拉取ServerList

回到ZoneAwareLoadBalancer，在其父类的构造方法中发现执行了一个初始化方法restOfInit()

enableAndInitLearnNewServersFeature,看名字大概是启用和初始化学习新服务实例。
这个serverListUpdater，又是在RibbonClientConfiguration配置文件中找到，为PollingServerListUpdater。
PollingServerListUpdater中实际执行的方法也为updateListOfServers。创建的是默认一秒钟之后执行，每隔30s执行，从eureka client刷新注册表到自己ribbon的LoadBalancer中。

updateListOfServers，就是更新serverList。com.netflix.niws.loadbalancer.DiscoveryEnabledNIWSServerList#getUpdatedListOfServers。获取eureka client中的注册表

八、spring整合后的ping机制

RibbonClientConfiguration中注册的是DummyPing，但是实现的通讯方法为空。
但是与eureka整合之后，在EurekaRibbonClientConfiguration中注册了一个NIWSDiscoveryPing。

Spring Cloud环境下，默认IPing组件不生效，只做简单的判断。因为集成了eureka。eureka自己有故障发现和服务实例摘除机制。在服务发生故障的时候，eureka server会发现然后摘除这个服务实例。eureka client会得到一个通知。eureka client本地的ribbon会每隔30s自动从eureka client本地获取注册表。

调用ping的地方：BaseLoadBalancer

特别注意pingIntervalSeconds，虽然在初始化的时候是10，但是在init方法中对这个值重新赋值，取的配置文件中的参数

九、负载均衡算法

  • RoundRobinRule

系统内置的默认负载均衡规范，直接round robin轮询，从一堆server list中，不断的轮询选择出来一个server，每个server平摊到的这个请求，基本上是平均的

  • AvailabilityFilteringRule

这个rule就是会考察服务器的可用性
如果3次连接失败，就会等待30秒后再次访问；如果不断失败，那么等待时间会不断边长
如果某个服务器的并发请求太高了，那么会绕过去，不再访问

  • WeightedResponseTimeRule

带着权重的，每个服务器可以有权重，权重越高优先访问，如果某个服务器响应时间比较长，那么权重就会降低，减少访问

  • ZoneAvoidanceRule

根据机房和服务器来进行负载均衡，说白了，就是机房的意思，看了源码就是知道了，这个就是所谓的spring cloud ribbon环境中的默认的Rule

  • BestAvailableRule

忽略那些请求失败的服务器，然后尽量找并发比较低的服务器来请求

  • RandomRule

随机找一个服务器，尽量将流量分散在各个服务器上

  • RetryRule

可以重试，就是通过round robin找到的服务器请求失败，可以重新找一个服务器
